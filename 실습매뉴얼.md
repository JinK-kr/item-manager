# AI로 물품 관리 웹앱 만들기 — 실습 매뉴얼

클로드 코드에게 말로 시켜서 **실제로 돌아가는 웹앱**을 만듭니다.
데이터베이스에 저장되고, 인터넷 주소로 남에게 보여줄 수 있고, AI로 물품을 등록합니다.

**완성 예시** — https://jink-kr.github.io/item-manager/

> 📌 이 문서는 두 가지 역할을 합니다.
> - **1~4부**: 학생이 따라 하는 실습 안내
> - **[5부: 현재 상태 인수인계](#5부-현재-상태-인수인계)**: 이미 만든 프로젝트를 새 대화에서 이어받는 데 필요한 정보
>
> **새 대화를 시작하셨다면 5부부터 읽으세요.**

---

# 1부. 시작 전에

## 무엇을 만드나

| 화면 | 하는 일 |
|---|---|
| 현황 대시보드 | 총 품목 수, 재고 부족, 카테고리 분포, 최근 변경 |
| 물품 목록 | 검색·정렬, 수량 조절, 삭제 |
| 물품 등록 | 폼으로 하나씩 + **말로 등록** (AI) |
| 엑셀 업로드 | 파일 하나로 수십 건 한 번에 |

## 준비물

- **클로드 코드** (설치되어 있어야 합니다)
- 구글 계정 (깃허브·Supabase 가입에 씁니다)
- 크롬 같은 최신 브라우저

## 돈이 드나요

**전부 무료입니다.** 카드 등록도 필요 없습니다.

| 서비스 | 쓰는 것 | 비용 |
|---|---|---|
| GitHub | 저장소 + Pages(웹사이트 공개) | 무료 |
| Supabase | 데이터베이스 + 서버 함수 | 무료 플랜 |
| OpenRouter | AI 모델 | `:free` 모델만 사용 |

## 작업 폴더

빈 폴더를 하나 만들고 **클로드 코드를 그 폴더에서 엽니다.**
폴더 이름에 한글이 있어도 됩니다. (예: `물품관리앱`)

# 2부. 준비 단계

## 준비 1. OpenRouter API 키 발급

AI 모델을 부르는 데 쓰는 열쇠입니다. **6단계에서만 필요**하니 나중에 해도 됩니다.

### 1-1. 가입하고 키 만들기

1. https://openrouter.ai 접속 → 구글 계정으로 로그인
2. 오른쪽 위 프로필 → **Keys** → **Create Key**
3. 이름은 아무거나 (예: `수업용`)
4. **Credit limit 을 0 으로** 걸어 두세요 ← 중요. 코드가 뚫려도 결제가 안 됩니다
5. `sk-or-v1-...` 로 시작하는 값이 나옵니다

> ⚠️ **이 창을 닫으면 다시 못 봅니다.** 준비 3의 Supabase 화면을 먼저 열어 두고,
> 새 탭에서 키를 만들어 바로 붙여넣는 순서가 편합니다.
> 놓쳤으면 옛 키를 지우고 새로 만들면 됩니다.

### 1-2. 무료 모델 이름 확인 ★

무료 모델은 **자주 바뀌고 사라집니다.** 실습 당일에 직접 확인하세요.

https://openrouter.ai/models 에서 **FREE** 로 거릅니다.
이름 끝이 `:free` 인 것만 무료입니다.

**두세 개를 골라 적어 두세요.** 하나가 막혔을 때 갈아탈 수 있게요.

```
예) 2026년 7월 기준
  google/gemma-4-26b-a4b-it:free                        ← 형식 강제 O (가장 안정적)
  google/gemma-4-31b-it:free                            ← json_object 까지
  nvidia/nemotron-3-nano-omni-30b-a3b-reasoning:free    ← 형식 강제 없음
```

> **고르는 요령**: 모델 상세 페이지에서 `structured_outputs` 를 지원하는 걸
> **적어도 하나는** 넣으세요. AI가 정해진 형식으로 답하게 강제할 수 있어서
> 훨씬 안정적입니다. 이게 없는 모델은 가끔 엉뚱한 형식으로 답합니다.

## 준비 2. 깃허브 저장소 만들기

만든 코드를 보관하고, 인터넷에 공개하는 곳입니다.

### 2-1. 저장소 만들기

1. https://github.com 가입 (구글 계정 가능)
2. https://github.com/new 접속
3. **Repository name**: `item-manager` (원하는 이름)
4. **Public** 선택 ← 나중에 웹사이트로 공개하려면 필요합니다
5. 아래 세 개는 **체크하지 마세요** — 나중에 충돌합니다
   - ~~Add a README file~~
   - ~~Add .gitignore~~
   - ~~Choose a license~~
6. **Create repository**

### 2-2. 클로드 코드에서 연결

```
이 폴더를 깃허브 저장소 https://github.com/[내아이디]/item-manager 에
연결해 줘. 아직 커밋할 건 없고 연결만 해 줘.
```

처음 푸시할 때 깃허브 로그인 창이 뜹니다. 로그인하면 다음부터는 자동입니다.

> **로그인 창이 안 뜨고 실패하면**: 깃허브 계정에 로그인한 상태로 브라우저를 열어 두고
> 다시 시도하세요. 그래도 안 되면 클로드에게 `"깃 자격 증명 상태를 확인해 줘"` 라고 물어보세요.

## 준비 3. Supabase 설정하기

데이터를 저장하는 곳입니다. 물품 목록이 여기에 쌓입니다.

### 3-1. 프로젝트 만들기

1. https://supabase.com 가입 (깃허브 계정으로 가능)
2. **New project**
3. 이름은 아무거나, 리전은 **Northeast Asia (Seoul)** 또는 가까운 곳
4. 데이터베이스 비밀번호는 적어 두세요 (실습에선 안 쓰지만 분실하면 곤란합니다)
5. 만들어지는 데 1~2분 걸립니다

### 3-2. AI 키를 Supabase에 보관 (준비 1을 했다면)

1. 왼쪽 메뉴 **Edge Functions** → **Secrets**
2. 두 칸을 채우고 **Save**

| 칸 | 값 |
|---|---|
| Name | `OPENROUTER_API_KEY` ← **철자 그대로** |
| Value | `sk-or-v1-...` (준비 1에서 받은 값) |

3. 아래 **Custom secrets** 목록에 이름이 뜨면 성공입니다. **값은 안 보이는 게 정상**입니다.

> ### 🔑 왜 `.env` 파일을 안 쓰나요
>
> 학생들이 가장 많이 하는 질문입니다.
>
> `.env` 는 **서버가 읽는 파일**입니다. 그런데 우리가 만드는 앱은 서버가 없습니다 —
> 브라우저에서만 돕니다. 그래서 `.env` 를 만들어도 **읽을 주체가 없습니다.**
>
> | 시도 | 결과 |
> |---|---|
> | `.env` 를 깃허브에 안 올림 | 배포에 안 올라감 → 앱이 키를 못 찾음 → **동작 안 함** |
> | `.env` 를 커밋 | **키가 그대로 공개됨** (봇이 몇 분 안에 긁어갑니다) |
> | JS 파일에 직접 씀 | **F12 누르면 다 보임** |
>
> 그래서 **Supabase 서버에 키를 맡깁니다.** 브라우저는 Supabase 함수에게만 말을 걸고,
> 그 함수가 대신 AI를 부릅니다. 키는 브라우저에도 깃허브에도 없습니다.
>
> ```
> 브라우저 ──▶ Supabase 함수 ──키 붙여서──▶ OpenRouter
>                    ▲
>            여기에만 키가 있음
> ```
>
> **이건 임시방편이 아닙니다.** 실제 회사들이 운영에서 쓰는 구조와 같습니다.
> 직접 서버를 굴리든 Supabase를 쓰든 원리는 똑같습니다.

### 3-3. 커넥터 연결 ★★ 가장 많이 막히는 곳

클로드 코드가 Supabase를 직접 만질 수 있게 연결합니다.

1. 클로드 코드에서 **디렉터리 → 커넥터** 를 엽니다
2. **Supabase** 를 찾아 설치하고 연결(로그인)합니다
3. **⚠️ 클로드 코드를 껐다가 다시 켜세요**

> 커넥터는 **대화가 시작될 때 한 번** 연결됩니다. 켜져 있는 상태에서 추가하면
> 그 대화에서는 안 잡힙니다.
>
> 증상: 커넥터 화면에는 *"이미 설치되어 있습니다"* 라고 나오는데
> 클로드는 *"Supabase 커넥터가 연결돼 있지 않습니다"* 라고 답합니다.
> **둘 다 맞는 말입니다.** 재시작하면 해결됩니다.

**연결 확인** — 새 대화에서 이렇게 물어보세요.

```
Supabase 커넥터가 연결됐는지 확인하고, 내 프로젝트 목록을 보여줘.
```

프로젝트 이름이 나오면 성공입니다.

# 3부. 만들기

## 1단계. PRD 만들기

**PRD** = 무엇을 만들지 미리 적어 두는 문서입니다.

바로 코드부터 만들면 AI가 이것저것 붙여서 산으로 갑니다.
**"무엇을 만들지"와 "무엇을 안 만들지"를 먼저 못 박아 두는 게 핵심입니다.**

### 프롬프트

```
사무실 공용 물품 관리 앱을 만들려고 해.

주요 기능은 물품 등록, 수량 늘리기와 줄이기, 물품 삭제
이 세 가지야.

사무실에서 공용 물품(볼펜, A4용지, 건전지, 물티슈 등)을
함께 쓰는 직원 10~30명을 위한 것이고,

누가 무엇을 얼마나 가져갔는지 기록이 안 남아서
재고가 떨어진 걸 필요한 순간에야 알게 되는 문제를 해결하기
위해 필요해.

어떻게 구현하면 좋을지
기술적인 방향과 함께
구체적인 PRD를 작성해 줘. PRD.md 파일로 만들어 줘.

아래 내용을 꼭 포함해 줘.

- 화면은 네 개다
  index.html(대시보드, 첫 화면) / items.html(물품 목록)
  register.html(물품 등록) / import.html(엑셀 업로드)

- 물품 정보: 물품 이름, 카테고리, 수량, 적정 재고량, 입고일,
  등록자 닉네임, 수정 시간

- 카테고리는 문구류 / 전자기기 / 청소용품 / 기타 네 가지로 고정

- 재고 부족 판정은 물품마다 다르게 한다.
  '적정 재고량'보다 지금 수량이 적으면 부족이고,
  부족 수량 = 적정 재고량 - 지금 수량 이다.

- 대시보드 위젯은 딱 네 개
  총 품목 수 / 재고 부족 / 카테고리별 분포 도넛 차트 / 최근 변경 내역

- 데이터는 Supabase에 저장한다

- 이번에 만들지 않을 기능도 명확히 적어 줘:
  로그인, 사진 업로드, 대여와 반납 기능은 만들지 않는다.
```

### 확인할 것

- `PRD.md` 파일이 생겼나
- **"만들지 않는 기능"** 항목이 들어갔나 ← 이게 있어야 나중에 안 흔들립니다
- 카테고리가 네 개로 고정됐나
- 적정 재고량 개념이 들어갔나

> **팁**: PRD를 읽어 보고 마음에 안 드는 부분은 그냥 말로 고치면 됩니다.
> `"카테고리에 예시 물품을 추가해 줘"` 처럼요.

## 2단계. 데이터베이스 만들기

### 프롬프트

```
PRD대로 Supabase에 items 테이블을 만들어 줘.

- 로그인이 없으니 누구나 읽고 쓸 수 있게 접근 정책(RLS)도 설정해 줘.
- 수정 시간은 값이 바뀔 때 자동으로 갱신되게 해 줘.
- 수량을 늘리고 줄이는 건 DB 함수로 만들어 줘.
  앱에서 "읽고 더하고 쓰기"를 하면 두 사람이 동시에 누를 때
  한 번이 사라지거든. 0 미만으로 내려가지 않게 막는 것도 거기서 해 줘.
- 만든 SQL은 supabase/schema.sql 파일로도 남겨 줘.
```

### 확인할 것

Supabase 대시보드 → **Table Editor** 에 `items` 테이블이 보이면 성공입니다.

컬럼이 이렇게 있어야 합니다.

```
id · name · category · quantity · target_quantity · received_at · owner · updated_at
```

> **여기서 배우는 것 두 가지**
>
> 1. **클로드가 SQL을 직접 실행합니다.** Supabase 화면에서 손으로 표를 만들 필요가 없습니다.
> 2. **동시성 문제.** 두 사람이 같은 물품의 `+`를 동시에 누르면 한 번이 사라질 수 있습니다.
>    이걸 DB 안에서 한 번에 처리하게 만드는 게 실무의 기본입니다.

## 3단계. 화면 만들기

### 프롬프트

```
PRD대로 앱을 만들어 줘.

[화면 구조]
- 왼쪽에 메뉴를 두고 오른쪽에 내용이 나오는 구조로 해 줘.
  좁은 화면(900px 이하)에서는 메뉴가 숨고 햄버거 버튼으로 열리게 해 줘.
  메뉴 내용은 한 파일에서만 만들어서 모든 화면이 같이 쓰게 해 줘.
- 메뉴: 현황 대시보드 / 물품 목록 / 물품 등록
- 첫 화면(index.html)은 대시보드로 해 줘.

[물품 목록 화면]
- 상단에 검색창 — 이름·카테고리·등록자를 한꺼번에 훑게 해 줘.
- 정렬 버튼 — 최신순 / 이름순 / 수량순 / 입고일순.
  켜진 버튼을 다시 누르면 방향이 뒤집히게 해 줘.
- 수량 옆에 + - 버튼과 삭제 버튼.
- 적정 재고량보다 적으면 "N개 부족" 배지를 붙여 줘.

[물품 등록 화면]
- 입고일은 오늘이 기본값이고 미래는 못 고르게 해 줘.
- 적정 재고량 기본값은 5로 해 줘.

[공통]
- 화면 밝기를 시스템/밝게/어둡게 중에 고를 수 있게 하고,
  고른 값은 다음에 열 때도 유지되게 해 줘.
- 휴대폰 화면(360px)에서도 안 깨지게 해 줘.

다 만들면 브라우저에서 실제로 눌러보고 잘 되는지 확인해 줘.
```

### 확인할 것

클로드가 알아서 브라우저를 띄워 확인합니다. 끝나면 직접 눌러 보세요.

- [ ] 물품을 등록하면 목록에 나타나나
- [ ] `+` `−` 로 수량이 바뀌나
- [ ] **수량 0에서 `−` 가 안 눌리나** ← 자주 빠지는 부분
- [ ] 삭제할 때 확인 창이 뜨나
- [ ] 검색이 되나, 정렬 버튼을 두 번 누르면 방향이 바뀌나
- [ ] 대시보드 위젯 네 개가 다 보이나
- [ ] 브라우저 창을 좁히면 메뉴가 숨나
- [ ] 밝기를 바꾸고 새로고침해도 유지되나

> **안 되는 게 있으면** 그냥 말하면 됩니다.
> `"수량 0인데 마이너스 버튼이 눌려. 고쳐 줘"`

## 4단계. 깃허브에 올리고 웹사이트로 공개하기

### 4-1. 캐시 문제 미리 막기 ★

**올리기 전에 이것부터 하세요.**

```
파일 주소에 ?v=1 같은 버전 번호를 붙여서
코드를 고쳐도 브라우저가 예전 파일을 쓰지 않게 해 줘.
앞으로 코드를 고칠 때마다 이 번호를 올려 줘.
```

## 6단계. 말로 등록하기 (AI 연동) ★

이 실습의 하이라이트입니다.
`볼펜 20개랑 물티슈 5개 들어왔어` 라고 적으면 알아서 두 건으로 나눠 줍니다.

**준비 1(API 키)과 준비 3-2(Supabase에 키 저장)를 먼저 끝내야 합니다.**

### 프롬프트

```
물품 등록 화면에서 말로 등록할 수 있게 만들어 줘.

[키 보관]
- OpenRouter를 쓰는데, API 키는 브라우저에 절대 넣지 마.
  Supabase Edge Function을 만들어서 거기서 부르고,
  키는 Supabase 시크릿의 OPENROUTER_API_KEY를 읽게 해 줘.
- .env 파일은 만들지 마.

[무료 모델만]
- 아래 세 개만 고를 수 있게 해 줘.
  [준비 1-2에서 적어 둔 모델 이름 세 개를 여기에]
- 호출하기 전에 OpenRouter에서 실제 가격을 받아와서
  0원이 맞는지 확인하고, 아니면 거절하게 해 줘.
- 화면에서 모델을 고를 수 있게 하고, 고른 값은 기억하게 해 줘.
- 모델마다 형식 강제 지원이 다르니까 거기에 맞춰 요청해 줘.

[안전장치]
- 바로 저장하지 말고 미리보기를 보여 준 다음 확인을 받아 줘.
- 카테고리는 AI가 추측한 값이니까 미리보기에서 드롭다운으로
  고칠 수 있게 해 줘. 이름·수량은 사람이 말한 그대로니까 고칠 필요 없어.
- 대화로는 등록만 되게 하고, 삭제나 수량 줄이기 요청은 무시하게 해 줘.
- "어제 볼펜 20개 들어왔어" 처럼 날짜를 말하면 입고일에 반영해 줘.
- AI가 실패해도 아래 직접 입력 폼은 그대로 쓸 수 있게 해 줘.

다 만들면 실제로 시험해 보고 결과를 알려 줘.
```

- [ ] `멀티탭 3개랑 HDMI 케이블 2개 들어왔어` → 두 건으로 나뉘나

> ### 왜 "다 보여주지 않는" 게 답인가
>
> 학생들이 배울 만한 판단입니다.
>
> - 대시보드는 **"지금 신경 쓸 일이 있나"** 를 보는 곳입니다.

>
> 대신 `부족 16건 · 채우려면 42개` 처럼 **전체 규모는 숫자로** 알려 줍니다.
> 5건만 봐도 "아 16건이구나" 가 바로 보입니다.

```

| `import_items(rows jsonb)` | 일괄 등록. 같은 이름이면 수량 합산. 한 트랜잭션 |

## 주요 설계 결정과 이유

새 대화에서 "왜 이렇게 돼 있지?" 할 만한 것들입니다.

| 결정 | 이유 |
|---|---|
| 수량 증감을 DB 함수로 | 앱에서 읽고-더하고-쓰면 동시에 누를 때 한 번이 사라짐 |
| `+`/`−` 눌러도 즉시 재정렬 안 함 | 누르던 줄이 튀어 올라가 다음 클릭을 놓침 |
| 합산 시 기존 카테고리·등록자 유지 | 이미 정해 둔 값을 입고 한 번으로 덮으면 안 됨 |
| `received_at` 은 `date` (timestamptz 아님) | 시간대 때문에 하루씩 어긋나 보임 |
| 미리보기에서 카테고리만 수정 가능 | 이름·수량은 사람이 말한 그대로, 카테고리만 AI 추측 |
| 재고 부족 위젯 5건만 표시 | 카드 높이 고정 + 대시보드는 요약하는 곳 |
| 엑셀 라이브러리 지연 로딩 | 900KB인데 대부분 사용자는 안 씀 |
| 무료 모델을 호출 직전 가격 조회로 검증 | 목록을 하드코딩하면 유료로 바뀔 때 못 막음 |

## 알려진 문제

1. **`gemma-4-31b:free` 를 고르면 실패해서 26b로 넘어갑니다.** 형식을 못 지켜서입니다.
   결과는 정상이지만 미리보기에 표시되는 "실제로 읽은 모델"이 다를 수 있습니다.
2. **다른 사람이 바꾼 내용은 새로고침해야 보입니다.** 실시간 반영이 없습니다.
3. **물품 정보 수정 기능이 없습니다.** 이름·카테고리·적정재고량을 나중에 못 고칩니다.
   엑셀로 다시 올리면 수량만 합산되고 나머지는 안 바뀝니다.
4. **일괄 삭제가 없습니다.** 그래서 엑셀·대화 모두 미리보기 확인을 반드시 거칩니다.
5. **기존 31건은 입고일이 전부 같고 적정 재고량이 전부 5입니다.**
   나중에 추가한 컬럼이라 기존 데이터에 의미 있는 값이 없습니다.
